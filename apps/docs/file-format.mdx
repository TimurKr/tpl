## Basic structure

```markdown
---
description: Optional human-readable description
---

Your prompt content goes here. Use {{variableName}} for variables.
```

The frontmatter is **optional**. A `.tpl.md` file with just content is valid.

## Variables

| Syntax | TypeScript | Behaviour |
|--------|-----------|-----------|
| `{{name}}` | `name: string` | Required string |
| `{{name:number}}` | `name: number` | Required number |
| `{{name:boolean}}` | `name: boolean` | Required boolean |
| `{{name:string[]}}` | `name: string[]` | Required array, joined with `, ` on render |
| `{{name\|default}}` | `name?: string` | Optional — uses default when omitted |
| `{{name:number\|0}}` | `name?: number` | Optional typed with default |

Types are declared right in the variable — no separate config that can drift out of sync.

```markdown
Summarize the following in {{wordCount:number}} words.
{{#if tone}}Write in a {{tone}} tone.{{/if}}
Focus on: {{focusArea|key decisions}}
```

## Conditionals

```markdown
{{#if note}}
**Note:** {{note}}
{{/if}}
```

Variables that appear only inside `{{#if}}` blocks are typed as optional booleans. If the same variable also appears as `{{note}}`, `{{note:number}}`, or another variable expression, TPL uses that declared type and marks it optional. The block is omitted when the variable is falsy or missing.

## Includes

Use `{{> name}}` to include another prompt by name:

```markdown
{{> basePersona}}

Now do the actual task using {{userInput}}.
```

- References resolve by **name** (camelCase or kebab-case), no path required
- Circular includes are detected — they produce a TypeScript error in the generated file rather than halting generation entirely

## Frontmatter

Only one field is currently used:

```yaml
---
description: Human-readable description of this prompt
---
```

The description appears as a JSDoc comment on the generated function and in IntelliSense.
