## The pipeline

```
.tpl.md files (co-located with your source code)
     ↓
  fast-glob discovers all files
     ↓
  gray-matter parses optional frontmatter
     ↓
  regex extracts {{variables}}, {{> includes}}, {{#if conditionals}}
     ↓
  include tree validated (circular detection, missing ref detection)
     ↓
  TypeScript interfaces + build functions generated
     ↓
  sibling .tpl.gen.ts files + lib/tpl.gen.ts manifest written to disk
```

Relative imports between generated `.ts` files use **`.js` specifiers** (e.g. `from "./welcome-email.tpl.gen.js"`) for strict ESM and TypeScript `Node16` / `NodeNext` compatibility.

## File naming → function names

The camelCase function name is derived from the file stem:

| File | `prompts` key | Build function |
|------|--------------|----------------|
| `welcome-email.tpl.md` | `welcomeEmail` | `buildWelcomeEmailPrompt()` |
| `search-query.tpl.md` | `searchQuery` | `buildSearchQueryPrompt()` |
| `classify.tpl.md` | `classify` | `buildClassifyPrompt()` |

Names must be unique across the entire project. Two files that produce the same name are both emitted into a single collision file with duplicate declarations, which TypeScript flags as errors — you'll see them immediately in the IDE.

Generated filenames preserve the template stem exactly: `welcome-email.tpl.md` generates `welcome-email.tpl.gen.ts`.

## The manifest exports

```typescript
// lib/tpl.gen.ts
export const prompts = {
  welcomeEmail: buildWelcomeEmailPrompt,
  searchQuery:  buildSearchQueryPrompt,
  // ...
} as const;

export function renderPrompt(name: keyof typeof prompts, vars = {}): string { ... }
```

`prompts` uses short keys for ergonomic call sites. `renderPrompt` enables dynamic dispatch when the template name comes from config or user input.

## Watch mode

`tpl watch` runs an initial generation on startup, then uses `chokidar` to watch for `.tpl.md` changes with a 50 ms debounce. On any change, it regenerates all files — the entire run typically takes under 100 ms even for large projects.
